## Drone dimensions function (what it does and why)

This repo centralises all “drone size” constants in `fleet_specs.py`, because drone size drives:

- the **atomic unit** (padded footprint for one drone),
- the **L0 cell size** (the bottom-layer graph resolution),
- and (with CORRIDRONE) safety buffers / geofence sizing.

### Where the dimensions come from

`fleet_specs.py` contains a small fleet dictionary:

- `FLEET_DATA["Standard"]`
- `FLEET_DATA["Oversize"]`
- `FLEET_DATA["H&B"]` (Heavy & Bulky)

Each tier has:

- `payload_kg`
- `dims_LWH_m` as a string like `"2.5 x 2.0 x 1.2"`

The dimensions represent an **enclosure / bounding box** (Length × Width × Height) needed to safely contain the drone.

### The actual “dimensions function”

The function you use elsewhere is:

- `get_drone_dims(drone_type, path=None) -> (length_m, width_m, height_m)`

Behaviour:

1) If `drone_fleet_specs.json` exists (generated by `calculate_fleet_specs()`), it reads the tier’s `dims_LWH_m` from JSON.
2) If not, it falls back to the built-in `FLEET_DATA`.
3) As a last resort, it returns `(1.0, 1.0, 0.6)`.

This design makes the simulation **reproducible** (JSON “locks” the specs) while still working out-of-the-box.

### Why use a bounding box and padding factors?

In motion planning, it is standard to reduce “a finite-size robot moving among obstacles” to “a point moving among inflated obstacles” by **fattening** objects using a Minkowski sum (configuration-space obstacle inflation). This is the core idea behind representing the drone by a footprint radius/box plus a clearance margin.

- The Algorithms / Computational Geometry reference notes: motion planning for polygonal robots “fattens each of the obstacles by taking the Minkowski sum … with the shape of the robot”, reducing planning to a point-robot shortest path problem.  
  Source: “Minkowski Sum”, *Algorithms* (Stony Brook / repository mirror). `https://www8.cs.umu.se/kurser/TDBAfl/VT06/algorithms/BOOK/BOOK5/NODE199.HTM`

In this project, we apply the same idea at grid-design scale:

- represent “one drone + safety margin” as a single **atomic unit** size, and then
- build a graph whose cells are multiples of that atomic unit.

### How the atomic unit is computed (and why)

The **atomic unit** is meant to be a *padded* “one-drone footprint” used for grid sizing.

#### Starting from scratch: a defensible padding rule

If the atomic unit is used as a “slot” size, the cleanest way to define padding is:

- start with the largest drone’s footprint side length \(S = \max(L, W)\)
- add a clearance margin that covers **position hold / tracking error** so two drones in adjacent slots don’t overlap

If two drones occupy adjacent slots of size \(A\), then the free gap between their footprints is:

\[
\text{gap} = A - S
\]

If the drone can deviate laterally by up to \(e\) (metres), then a simple conservative “no-overlap” condition is:

\[
A - S \ge 2e \quad \Rightarrow \quad A \ge S + 2e
\]

**Choosing \(e\) from real drone specs (research-backed):**

DJI publish “Hovering Accuracy (windless or breezy)” figures that bound how accurately the aircraft can hold position:

- Horizontal: **±0.3 m (Vision System enabled)**; **±1.5 m (N‑mode with GPS)**; **±0.1 m (RTK)**  
  Source: DJI Matrice 30 Series specs. `https://enterprise.dji.com/matrice-30/specs`
- The same order-of-magnitude is also listed for consumer drones (e.g. Mini 2): Horizontal **±0.3 m (Vision)**, **±1.5 m (GPS)**  
  Source: DJI Mini 2 specs (mirror). `https://drdrone.com/pages/dji-mini-2-technical-specifications`

For **L0 / low-altitude** operations in cities, it is reasonable to assume **vision-assisted positioning is available much of the time** (and in any case, our CORRIDRONE layer handles larger-scale uncertainties). So we set:

- \(e = 0.3\ \text{m}\) (vision-enabled horizontal hold bound)

Then:

\[
A = S + 2e = S + 0.6
\]

For the largest drone tier in this model (H&B), \(S = 2.5\ \text{m}\), so:

\[
A = 2.5 + 0.6 = 3.1\ \text{m}
\]

As a **ratio**, that is:

\[
\text{padding factor} = \frac{A}{S} = \frac{3.1}{2.5} = 1.24
\]

**Final recommended padding factor (rounded): `1.25`.**

#### Why not size atomic padding for GNSS-only (±1.5 m)?

If we used \(e = 1.5\ \text{m}\) (GPS-only), we’d get:

\[
A = S + 3.0 \quad \Rightarrow \quad \text{factor} \approx 2.2
\]

That would make the grid far coarser, mainly to compensate for navigation uncertainty that should be handled by the **operational safety layer** (CORRIDRONE) rather than by the discretisation unit.

#### What the code currently does

Today, `fleet_specs.py` computes the atomic unit as:

- \(A = \max(L,W) + 2e\)

with a default lateral padding \(e \approx 0.30\ \text{m}\), which is consistent with the “start from scratch” rule above and yields a factor \(\approx 1.24\) (rounded to **1.25**) for the H&B drone.

### How L0 cell size is computed (and why)

`fleet_specs.py` defines:

- `L0_BUCKET_FACTOR = 4`
- **L0 cell** (metres) = `4 * atomic_unit`

This “bucket” design is used because:

- an atomic unit is “just enough” space for one padded drone,
- but a graph node at atomic resolution would make pathfinding expensive,
- so L0 groups \(4 \times 4 = 16\) atomic units into one navigation cell, giving a practical resolution/complexity trade-off.

In short:

- **atomic unit** encodes drone footprint + clearance
- **L0** encodes “lowest-resolution navigation graph that is still plausible for terminal manoeuvring”

### How this connects to CORRIDRONE

The CORRIDRONE geofence sizing (`corridrone.py`) takes:

- speed (m/s)
- `get_drone_dims()` output

and uses the drone’s physical size as the “body radius” term, then adds:

- response distance,
- braking distance,
- navigation uncertainty,
- wind drift during response+braking.

So the *same* dimension source drives both grid sizing (static spatial discretisation) and geofence sizing (dynamic safety buffer).

### Is atomic-unit padding still required if CORRIDRONE geofencing exists (especially on L0)?

**Not for safety, if you actually enforce CORRIDRONE.**

- CORRIDRONE is the *operational* safety buffer: it already accounts for wind drift, response time, braking distance, and navigation uncertainty, and it scales with speed.
- Atomic/L0 sizing is the *discretisation* choice: it ensures the grid resolution isn’t smaller than the drone footprint and keeps pathfinding tractable.

So you should avoid “double counting”:

- Keep the atomic unit as a **modest geometric footprint** (at least \(S=\max(L,W)\), optionally with small padding like the 1.25 above).
- Treat CORRIDRONE as the thing that provides the **real separation buffer** between active aircraft, including on L0.

### Why we do NOT “bake CORRIDRONE into the atomic unit”

Your proposed rule (paraphrased) is:

> atomic unit should include the CORRIDRONE geofence at the lowest (L0) speed, and then we take `max(padding, CORRIDRONE)` so we don’t double count.

That rule would be correct **only if** the atomic unit represented “one exclusive occupancy slot per simultaneously-flying drone”, i.e. if the model assumed:

- at most 1 drone per atomic unit at a time, and
- neighbouring atomic units must guarantee CORRIDRONE separation by construction.

But this project’s grid is **not** an “occupancy lattice”. It is a **routing discretisation**:

- drones follow graph edges; multiple drones can traverse the same cell/edge at different times
- safety separation is an *operational constraint* (CORRIDRONE / conflict checking), not a property guaranteed by cell size

If you bake CORRIDRONE’s full operational buffer into the atomic unit, it quickly becomes enormous because CORRIDRONE includes:

- response distance \(v\tau\),
- braking distance \(v^2/(2a)\),
- wind drift during \(\tau + v/a\),
- navigation uncertainty.

Even at low speed, the wind-drift + uncertainty terms can dominate. For example, with the CORRIDRONE defaults in `corridrone.py`:

- \(v = 1\ \text{m/s}\), \(\tau = 1\ \text{s}\), \(a = 3\ \text{m/s}^2\), wind \(w=5\ \text{m/s}\)
- wind drift term \(w(\tau+v/a) \approx 5(1+0.33)=6.7\ \text{m}\)
- plus position uncertainty (default 5 m)

This already puts the geofence radius into the **~10–20 m** range after adding body size and response/brake distances. If you then require center-to-center spacing \(\ge 2r\), the “atomic unit” becomes **tens of metres**, and L0 (4× atomic) becomes **hundreds of metres** — which destroys the grid resolution you need for terminal routing.

So the correct modelling separation is:

- **Atomic unit**: choose for *geometric plausibility + computational tractability* (metres-scale).
- **CORRIDRONE**: apply as an *operational safety envelope* (often tens-of-metres-scale), independent of cell size.

### Do different layer speeds force the drone’s outer dimensions to change?

Not necessarily.

There are two different “dimensions” concepts:

1) **Outer envelope (L×W×H)** — rotor span / frame / payload bay / prop guards. This usually changes slowly with product class and payload needs.
2) **Mass & battery energy (kg, Wh)** — this *does* change with mission design and speed because energy consumption depends on speed.

Multirotor energy consumption is speed-dependent:

- Hwang et al. (2018) model forward flight and show drag force \(D \propto U^2\) and the power to overcome drag \(D \times U \propto U^3\). They also show an optimal speed exists (U‑shaped required power vs speed).  
  `https://www.mdpi.com/1996-1073/11/9/2221`

So if you change per-layer speeds, you should treat the impact primarily as:

- **battery energy requirement changes** (Wh needed for the worst mission profile),
- hence **battery mass changes**,
- which may change performance limits (acceleration, climb rate) in a deeper model.

But it does *not* automatically mean the drone’s **external L×W×H** must be recomputed every time the speed schedule changes. In many designs the battery is an internal module within a fixed frame.

#### Practical modelling choice for this EPQ

- Keep `dims_LWH_m` as a **class-level envelope** (Standard / Oversize / H&B).
- Treat speed schedule as affecting **operational parameters** (energy used, separation buffers), not the envelope.

If you want to be more rigorous later, you can introduce an iterative design loop:

1) pick a speed schedule by layer
2) compute worst-case mission energy for that schedule (using a power-vs-speed model)
3) choose battery Wh (and battery mass)
4) re-check performance constraints (e.g. decel/climb used in CORRIDRONE)
5) iterate until stable

That’s a natural “next model fidelity” step, but it’s overkill for the current grid + routing prototype.
